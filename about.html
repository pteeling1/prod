Purpose

This tool was built to help infrastructure planners quickly translate workload requirements into validated hardware configurations. Whether you're exploring datacenter or edge deployments or simply refining a single instance, it gives you clarity, flexibility, and presentation-ready outputs.

How to Use

The tool supports two modes:

  • Automated Sizing Mode: Click Run Sizing to input workload requirements either by Virtual Machine specifications or infrastructure totals — choosing between total CPU cores or total CPU GHz. You can also specify growth percentage and high availability level. These guide the engine’s recommendations.
  • Manual Configuration Mode: Adjust sliders and dropdowns to customize machine count, disk layout, memory size, and more. Ideal for field overrides or exploratory planning.

After sizing, you’ll see a visual summary, a bar chart comparing requirements vs. provisioned capacity, and post-failure capabilities. You can export results to PowerPoint for sharing or convert to PDF externally if needed.

Design Philosophy

This tool is built around clarity, operational realism, and deliberate constraint. We’ve made specific design choices to reflect best practices and platform optimizations:

  • Storage: We exclusively provide single-tier flash storage — NVMe or SSD — with no hybrid disk scenarios. HDD-based caching tiers or parity layouts are not supported. For resiliency, we default to 3-way mirror for instances larger than two machines, and 2-way mirror for one- and two-machine deployments.
  • Compute: We prioritize minimizing machine and core count where possible. The sizing engine automatically selects the highest-clocked Intel® Xeon® Gold 6000 series processor that satisfies the core requirement — balancing performance with efficiency.
  • Instance Sizing: In automated mode, if a workload requires more than seven machines, the engine will automatically split the configuration into multiple instances of seven or fewer. This reflects a known design preference in Azure Local, where smaller instance sizes tend to behave more predictably.
  • Transparency: All sizing logic — including disk reserve calculations, post-failure modeling, and memory layout — is exposed and traceable. Manual override mode lets you explore configurations freely, while the chart remains anchored to your original requirements for clarity.
  • Reference: All the how-to information, why the sizing engine works the way it does is based on Microsoft documentation, described here Azure Local on Microsoft website.

Fun Facts

In mid-July of 2025, my friend Greg Reeder sent me a Teams message: "They cancelled that meeting again for the 70th time — how are we ever going to get a good tool?" That was the spark. That same evening, after hours, I started building a new sizing tool from scratch.

I’m not a developer, and I’m not particularly good at writing code — but I figured with AI tools, I might be able to get something working quickly. And that turned out to be completely true. Within minutes, a strawman design of a sizing engine was generated by Copilot, Microsoft’s AI companion. From there, Copilot helped write the logic, debug edge cases, refine disk and memory calculations, and even draft the visual layout and export formatting.

But as with all AI tools, they don’t replace human experience — they amplify it. Over the following months, I worked closely with a team of experts to refine and evolve the tool into what you see today. Huge thanks to:
