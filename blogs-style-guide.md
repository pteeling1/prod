# Teeling Cloud Courant - Writing Style Guide

## Tone & Voice
- Professional but conversational
- Authoritative without being condescending
- Focus on practical reality over marketing claims
- Use "you" language to address readers directly
- Include specific facts, dates, and technical details

## Structure
- **Headline**: Direct, benefit-driven (e.g., "Making the Case for 100 GbE Networking")
- **Subtitle**: Context or key takeaway (e.g., "Why dense NVMe nodes change the networking conversation sooner than most designs expect")
- **Opening**: Hook the reader with a question or problem statement
- **Body sections**: Use clear section headers (UPPERCASE WITH UNDERLINE)
- **Bulleted lists**: For comparisons, requirements, or actionable items
- **Closing**: End with practical guidance or a summary statement

## Technical Depth
- Assume readers are infrastructure architects, not beginners
- Use real VMFleet data, support dates, and technical specs
- Avoid generic cloud marketing language
- Support claims with specifics (e.g., "roughly 12 NVMe drives per node" not "many drives")
- Include thresholds and decision points (e.g., "≤ 1 millisecond round-trip latency")

## Key Topics to Emphasize
- Architecture decisions based on data, not trends
- Real-world constraints and tradeoffs
- Microsoft's actual support policies and timelines
- Hybrid cloud as a legitimate architectural choice
- Azure Local specifics (formerly Azure Stack HCI)

## Example Opening Patterns
- "Still on 22H2? Running a stretched cluster? Or sitting on 23H2 hoping it buys you time?"
- "As NVMe becomes the default choice for modern hyperconverged infrastructure, one question comes up repeatedly..."
- "All-NVMe Storage Spaces Direct and Azure Local fundamentally change where performance bottlenecks appear."

## Quote and Sidebar Usage
- Use sparingly for emphasis
- Pull quotes from industry analysts or Microsoft guidance when relevant
- Sidebar boxes for decision matrices or quick reference tables

## Typography Notes
- Use em‑dashes (—) not hyphens
- Use proper quotation marks ("") not straight quotes
- Numbers: spell out "two" through "nine", use numerals for 10+
- Acronyms: Spell out first use (e.g., "Azure Local (formerly Azure Stack HCI)"), then use shorthand

## Article Length
- 2,000–4,000 words for deep dives
- Focus on completeness over brevity
- Divide long pieces into clear sections with headers
